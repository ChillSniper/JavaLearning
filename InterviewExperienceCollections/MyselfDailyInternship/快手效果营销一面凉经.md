MySQL

- 能说一下mysql索引的结构吗？

> MySQL(InnoDB) 索引结构是B+树
> B+树非叶子结点只存储键和指针，所有数据存储在**叶子结点**，叶子结点通过双向链表连接，适合范围查询

- 再具体点，就是B+树里面的结构大概是啥样的。

> 一个B+树节点对应一个**页**，大小是16KB
> 非叶子节点：存储多个key+指向子节点的指针（不存数据）
> 叶子结点：存储key+row pointer（主键索引下存整行记录）；同时叶子结点之间使用双向链表连接，支持范围查询和顺序读取

- 索引覆盖的概念有了解过吗？

> 查询所需的列全部能在一个索引里找到，不需要回表到主键索引查整行数据，这就是**索引覆盖**
> 比如 select id, name from user where name = 'a'，如果name上有索引，那么就有可能覆盖索引

- 在你有处理过mysql的慢查询吗？

Redis

- 我看你写到了解穿透、雪崩和击穿，那你能分别说说这三种情况吗？

> - 缓存穿透：查询的数据不存在，Redis和db都查不到；解决方案：缓存空值或者使用布隆过滤器
> 缓存击穿：某个热点key过期瞬间，大量请求同时打到DB；解决方案：互斥锁，或者使用逻辑过期（前者保证的数据一致性，后者保证的高可用性）
> 缓存雪崩：大量key同时过期，导致大量查询到达数据库；解决方案：过期时间随机化、分布式限流

- 那我们再聊聊分布式锁，我看你也写分布式锁了，那怎么用redis实现分布式的锁？

> 最常用的方式：setnx+expire（原子set命令：SET KEY VALUE NX EX）
> 释放锁用lua，保证VALUE对比 + 删除的原子性
> 为了解决锁误删：使用唯一标识（UUID+ThreadID）

- 那你能大概说一说你的理解，就是为什么redis比较适合做分布式锁吗？

> - redis是单线程+内存操作，天然具备原子性
> - 读取写入极快，锁开销很低
> - 支持过期时间自动释放死锁
> - 支持lua，能把检查+删除做成原子操作
